> この記事は、前身プロジェクト「Saru」の開発記ブログ([Saru Blog](https://ko-chan.github.io/saru-blog/))の最終回として書いたものを、後継プロジェクト flowercrown のブログ第 1 号として公開するものです。

## この記事で得られること

- ボトムアップ開発が「ログインできるだけのシステム」を構造的に生み出す理由
- speckit のような厳格なワークフローが、かえって失敗を加速させる条件
- 代わりに採るべき「縦スライス(walking skeleton)」という作業単位

## はじめに ― 結末から書く

前身ブログでは、マルチテナント型サブスクリプション管理システム「Saru」を1人で作る開発記として、第1回から第4回まで公開してきた。その締めくくりは、結末から書く。

**Saru は畳んだ。** 9 ヶ月で 1,500 本以上の PR をマージし、CI は常に緑、デモ環境へのデプロイも通っていた。それでも、ユーザーがログインした先で、製品として何ひとつ完結できなかった。商品を設定し、見積を作り、注文し、決済し、サブスクリプションが始まる ― その一連の流れ(ジャーニー)が、最後まで1本も通らなかった。

[第1回の記事](https://ko-chan.github.io/saru-blog/posts/saru-mulit-tenant-subscription-system-1/)で、私は speckit を中心にした開発ワークフローを「複雑なシステムを1人で作り切る方法」として紹介した。結論から言うと、あれは間違っていた。ただし「speckit が悪かった」という単純な話ではない。失敗の構造は、もっと深く、もっと地味なところにあった。この記事は、その解剖だ。

## 何ができて、何ができなかったか

| 領域 | 状態 |
|------|------|
| 4ポータルのUI | できた |
| 認証(Keycloak / WebAuthn / OTP) | できた |
| アカウント・ユーザーのCRUD | できた |
| RLS によるテナント分離 | できた |
| E2Eテスト・CI/CD パイプライン | できた |
| 商品設定 → 見積 → 注文 → 決済 → サブスク開始 | **1本も通らない** |

左の列はすべて「緑」だった。ユニットテストも通る。デプロイも通る。それでも最下行 ― 製品の存在理由そのもの ― が動かない。

ここで気をつけたい言い方がある。「土台はできていて、あとは統合が残っているだけ」。これは**慰めであって、事実ではない**。むしろ、この言い方こそが、振り返るべき間違いそのものだった。どのジャーニーも通らないドメイン層、どの画面も使わないスキーマ、どの動作するフローも保護していない RLS ― それは「できている」とは言わない。動かしていないコードは資産ではない。善意で書かれた負債だ。

## 662 という化石

開発を畳んだ時点で、`specs/` ディレクトリには **662 個**のスペックフォルダが積み上がっていた。speckit の `specify` は、1サイクルにつき1フォルダを作る。662 回、仕様を書き、計画し、タスクに割り、実装し、検証した。そのほとんどで、CI は緑になった。

662 回まわして、出来上がったのは、ログイン画面の先へ進めないシステムだった。この乖離が、この記事で説明したいことのすべてだ。

## 根本原因は speckit ではない ― 「ボトムアップ」だ

Saru の作り方は**ボトムアップ**だった。スキーマを作り、次にドメイン層、次にインフラ、次に API、UI と統合を最後に回す。

ソフトウェアは、縦に層が積み重なった構造をしている。

```
  UI         ← ユーザーが実際に触る面(一番「上」)
  API
  UseCase
  Domain
  Schema/DB  ← データの土台(一番「下」)
```

ボトムアップとは、この一番下の Schema から着手して、上の UI へ向かう作り方のこと。「Schema を作る」というタスクで製品全体ぶんのテーブルを作り、次の「Domain を作る」で全体ぶんのドメインモデルを作り……と進む。1,500 PR を消化した時点の絵は、こうなる。

```
  UI        ░░░░░░░░  ← 最後の工程。未完成のまま時間切れ
  API       ████████
  UseCase   ████████
  Domain    ████████
  Schema    ████████  ← 最初の工程
```

完成した層は4枚。だが、ユーザーが端から端まで歩けるジャーニーはゼロ。これは「たまたま統合を忘れた」のではない。ボトムアップは、この結果を**構造的に作る**。

1. **統合は必ず最後に来る。そして、最後の工程は切られる工程になる。** スケジュールが押せば削られるのは最後の工程だ。その最後の工程が「全層を1本につなぐ」工程なので、削られた瞬間、製品はゼロになる。
2. **各層は「単体で」緑にできる。** スキーマのテストも、ドメインのテストも、それぞれ単独で通る。だから PR は毎回緑になり、毎回「進捗した」感触が出る。だが、ユーザーが触るものは一度も完成していない。
3. **使われない層を作り込む。** 「層を完成させる」発想だと、後のジャーニーが使わないカラム・enum・型まで、先回りして作ってしまう。Saru は、UI に出ているのに何も起こさないトグルや、保存されるだけで誰も読まない設定値を、大量に出荷した。すべて「層を完成させた」副産物だ。

理由を突き詰めると、一行になる。**ユーザーは「層」を触らない。層を縦に貫く1本のジャーニーを触る。** 層が何枚完成しても、貫通するジャーニーが1本もなければ、製品は存在しない。

公平を期して言えば、ボトムアップは主犯であって、唯一の犯人ではない。MVP のスコープを (商品タイプ7種類、取引パターン5つ……と) 先回りで広げすぎたことも、ソロ運用に対する見積もりの甘さも、共犯だった。ただ、共犯を全部除いたあとに残る最深の単独要因は何かと問えば、作業単位の形 ― ボトムアップ ― だ。スコープを絞っていても、ボトムアップなら結局ログインの先で何も動かない。逆にスコープを誤っても、縦スライスならせめて「歩く骨格」が残る。だからこの記事はそこを取り出して書いている。

## なぜ speckit は事態を「悪化」させたのか

では、なぜ「speckit が悪い」という話ではないのか。

speckit(spec → plan → tasks)は、仕様を明示し、曖昧さを潰し、計画を立ててから実装する、れっきとした規律だ。規律そのものは良いものだ。問題は、その規律を**間違った作業単位に当てた**ことにある。

層の単位で spec → plan → tasks → implement を回せば、層の単位で、緑のスペックと緑の PR が量産される。662 個のスペックは、その「速く回る間違い」の化石だ。

ここが、この記事で一番伝えたい点だ ― **間違った単位に rigor(厳密さ)を当てると、偽の自信を、より速く製造する。** 規律のないボトムアップなら、どこかで「あれ、何も動いていないぞ」と気づけたかもしれない。規律のあるボトムアップは、緑の PR を量産しながら、その気づきの瞬間を、最後まで先送りにする。速度は、形の間違ったプロセスに乗ると、製品ではなく負債を、速く生む。

speckit に自分で被せて作った QA コマンドが、それを象徴している。`qa.verify-rls`、`qa.verify-migration`、`qa.verify-api`、`qa.verify-design`、`qa.verify-impl` ― 検証の語彙そのものが、すべて「層」の名前だった。`qa.verify-journey` は無かった。検証の仕組み全体が「層を1枚ずつ認定する」ためにチューニングされていて、一番大事な「ユーザーがゴールデンパスを歩けるか」を測るゲートだけが、最初から存在しなかった。

## 正しい単位 ― 縦スライス(walking skeleton)

作り直しで採った方法は、これだ。**縦スライス。** 1本のジャーニーを選び、そのジャーニーが必要とするぶんだけ ― それ以上は1ミリも作らずに ― Schema / Domain / API / UI を、同時に、薄く貫通させる。

```
            slice1  slice2  slice3
  UI        ██████  ░░░░░░  ░░░░░░
  API       ██████  ░░░░░░  ░░░░░░
  UseCase   ██████  ░░░░░░  ░░░░░░
  Domain    ██████  ░░░░░░  ░░░░░░
  Schema    ██████  ░░░░░░  ░░░░░░
```

最初のスライスが終わった時点で、薄くても1本のジャーニーが、実環境で歩ける。これが Alistair Cockburn の言う walking skeleton(歩く骨格)だ。骨格は初日から歩き、あとはジャーニーを1本ずつ足して肉付けしていく。ボトムアップが「水平の層」を積むのに対し、こちらは「垂直の1本」を通す。だから縦スライスと呼ぶ。

| | ボトムアップ | 縦スライス |
|---|---|---|
| 作業単位 | 層(横) | ジャーニー(縦) |
| 1単位の「完了」 | その層が緑 | そのジャーニーが実環境で歩ける |
| 統合のタイミング | 最後(=切られる) | 毎回(スライスごと) |
| 進捗の指標 | 完成した層の数 | 歩けるジャーニーの数 |
| 同じ量の作業をこなした後 | 緑の層は多い／製品はゼロ | 薄いが、歩ける製品 |

そして、マージの合格条件を入れ替えた。「CI が緑」ではなく、**「この変更のあとも、実環境でゴールデンパスがまだ歩けること」**。緑のユニットテストはもちろん必要だが、それだけでは十分条件でも合格ゲートでもない。最上位のテストはジャーニーそのもの ― これだけが製品の合格を判定する。

## 後継プロジェクトへ持っていった、たった1つのルール

Saru を畳んで作り直しを始めたとき、最初に決めたルールは、1つだけにした。

> **縦スライスで作る。ボトムアップは禁止。**

「スキーマを作る」というタスクは無い。「ドメインモデルを作る」も、「API を作る」も無い。あるのはジャーニーだけだ。どのタスクも、DB・ドメイン・API・UI を、その1本のジャーニーが必要とするぶんだけ触り、そのジャーニーが実環境で歩けたところで終わる。

「層が単体で動く」は、何の意味も持たない指標だ。意味のある数字は、実際のユーザーが完走できるジャーニーの数 ― それ1つだけ。Saru の 662 スペックと 1,500 を超える緑の PR は、その教訓のために払った授業料だった。

## おわりに ― このブログについて

前身のシリーズは、複雑なものを1人で作り切る方法を共有するつもりで始めた。結果として共有できる一番価値のある知見が、皮肉にも、失敗の構造そのものになった。

speckit も、RLS も、CI ゲートも、並列エージェントによる開発速度も、どれも良い道具だ。だが、良い道具をいくら積み上げても、作業単位の形が間違っていれば、製品にはならない ― むしろ、間違いを、速く、きれいに、積み上げてしまう。

道具を選ぶ前に、単位を疑う。それが、Saru 開発記の結論です。

後継プロジェクトは **flowercrown** と名付けた ― 別人になったわけではなく、Saru で学んだことを土台に、縦スライスで作り直している同じ取り組みの、続きだ。このブログ(Flowercrown Blog)は、その作り直しの記録になる。
